1.¿Cuál es el bug raíz y en qué archivo/función está?
El bug raíz está en el archivo crypto/algif_aead.c del kernel Linux, en la función _aead_recvmsg(). En 2017 alguien optimizó esta función para que usara el mismo espacio de memoria como entrada y salida al mismo tiempo. El problema es que otro archivo, crypto/authencesn.c, escribe 4 bytes fuera del área que le corresponde. Cuando estas dos cosas se combinan, esos 4 bytes caen en la memoria del kernel donde están guardados los archivos del sistema.
2.¿Por qué el write a dst[assoclen + cryptlen] es peligroso?
El kernel tiene reglas sobre hasta dónde puede escribir cada función. La posición dst[assoclen + cryptlen] está fuera del límite permitido. Normalmente esto no causaría problema, pero con el cambio de 2017, en esa posición están las páginas en RAM de archivos reales del sistema como /usr/bin/su. El atacante puede controlar qué archivo modificar, en qué posición exacta y qué valor escribir. Con eso puede meter código malicioso en un archivo del sistema sin necesitar permisos de administrador.
3.¿Por qué el exploit es "stealthy" (no modifica el archivo en disco)?
El kernel guarda copias de los archivos en RAM para no tener que leer el disco cada vez, esto se llama page cache. Copy Fail escribe directamente en esa copia en RAM pero el kernel no sabe que fue modificada porque nunca la marca como dirty. Si la página no está marcada como dirty, el kernel no la escribe de vuelta al disco. Por eso el archivo en disco queda igual y programas como sha256sum no detectan nada raro. Pero cuando alguien ejecuta /usr/bin/su, el kernel usa la versión en RAM que ya está corrompida.
4.Conecta esto con lo que vimos en clase: page cache, chmod, setuid, inodos
Sobre los conceptos de clase: el page cache es exactamente lo explicado, el kernel guarda archivos en RAM para ir más rápido y Copy Fail abusa de eso. El binario /usr/bin/su tiene permisos especiales con chmod 4755 que le permiten ejecutarse como root aunque lo lance un usuario normal, Copy Fail mete shellcode ahí y cuando se ejecuta corre como root. El inodo guarda información del archivo como permisos y fechas pero no el contenido en RAM, como Copy Fail solo toca el page cache el inodo no cambia y nadie nota que algo pasó.
5.¿Qué aprendiste sobre cómo múltiples cambios "razonables" pueden crear un bug grave?
Copy Fail no fue culpa de un solo programador. Fueron tres cambios separados en 6 años: en 2011 se agregó authencesn para IPsec y usaba el buffer de destino como espacio temporal, tenía sentido en ese momento. En 2015 se expuso el subsistema criptográfico a usuarios sin privilegios con AF_ALG, era seguro porque usaba memoria separada. En 2017 se optimizó el código para usar la misma memoria como entrada y salida y nadie conectó esto con lo de 2011. Cada cambio por separado parecía correcto, nadie vio el sistema completo. Esto me enseñó que en sistemas grandes los bugs más peligrosos no son errores obvios sino el resultado de varias decisiones que parecían buenas por separado.
